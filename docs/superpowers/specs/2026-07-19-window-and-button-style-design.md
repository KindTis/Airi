# 창 크롬과 작업 버튼 스타일 통일 설계

- 작성일: 2026-07-19
- 상태: 사용자 승인 완료
- 대상: 메인 창, 메타데이터 편집창, 썸네일 선택창

## 1. 목표

- 메인 창의 회색 네이티브 타이틀바를 현재 다크 네이비 화면과 어울리는 커스텀 타이틀바로 바꾼다.
- 메타데이터 편집창과 썸네일 선택창에서 네이티브 타이틀바와 프레임을 제거한다.
- 사용자가 직접 누르는 텍스트 작업 버튼의 형태와 상태 표현을 `Missing Metadata Only` 버튼 기준으로 통일한다.
- 주요 동작은 기존 파란색 계열, 보조 동작은 기존 어두운 계열로 구분한다.

## 2. 범위

### 포함

- `MainWindow`에 기존 `CustomWindowStyle` 적용
- 메인 타이틀바 배경·외곽선을 앱의 다크 네이비 팔레트로 지정
- `MetadataEditorWindow`와 `ThumbnailSelectionWindow`를 `WindowStyle="None"`인 borderless 창으로 변경
- 공통 형태를 가진 주요·보조 작업 버튼 스타일 정의 및 적용
- 관련 WPF 회귀 테스트, 전체 테스트, Debug 빌드, graphify 갱신

### 제외

- 검색 입력창 안의 원형 지우기 버튼
- 타이틀바의 최소화·최대화·닫기 버튼
- 날짜 선택기와 다른 컨트롤 템플릿 내부의 버튼
- 창 크기, 배치, 문구, 명령, 클릭 처리 변경
- borderless 다이얼로그용 새 닫기 버튼 또는 드래그 기능 추가
- 새로운 색상 설정, 테마 시스템 또는 버튼 컴포넌트 계층 추가

## 3. 선택한 접근

기존 `Themes/Controls.xaml`의 `CustomWindowStyle`을 재사용하고, 앱 리소스에 작업 버튼용 공통 스타일만 추가한다. 각 창에 템플릿을 복제하거나 앱의 암시적 기본 `Button` 스타일을 전역 교체하지 않는다.

전역 기본 스타일 교체는 날짜 선택기 등 범위 밖의 내부 버튼에도 영향을 주므로 제외한다. 메인 창을 완전한 커스텀 프레임으로 새로 만드는 방식과 DWM API로 네이티브 타이틀바 색을 바꾸는 방식도 기존 구현 재사용보다 코드와 플랫폼 의존성이 커서 제외한다.

## 4. 시각 규칙

### 팔레트

- 셸 배경: `#0F111C`
- 타이틀바·패널: `#151828`
- 보조 버튼 배경: `#1E2234`
- 보조 버튼 테두리: `#2E3350`
- 주요 버튼 배경·테두리: `#3B82F6`
- 기본 텍스트: `#FFFFFF`

새 색상 체계를 만들지 않고 현재 세 창에서 이미 사용 중인 값을 재사용한다.

### 형태와 상호작용

- 모든 대상 작업 버튼은 14px 둥근 모서리와 1px 테두리를 사용한다.
- 현재 버튼별 너비, 높이, 여백과 텍스트는 유지한다.
- 주요·보조 버튼은 같은 템플릿을 공유하고 기본·hover·pressed·disabled 상태의 색상만 역할에 맞게 다르게 둔다.
- 키보드 포커스와 `IsDefault` 동작은 기존 WPF 동작을 유지한다.
- `Missing Metadata Only`의 선택 상태 표현은 그대로 유지하며 새 작업 버튼 형태의 기준으로 사용한다.

레이아웃과 글꼴은 변경하지 않는다. 이 작업의 시각적 특징은 기존 화면에 이미 있는 둥근 네이비 버튼과 파란색 활성 색상의 일관된 반복이다.

## 5. 버튼 역할 분류

### 주요 버튼 — 파란색 계열

- 메인 창: `Random Play`, `Fetch Metadata`
- 메타데이터 편집창: `파일 선택`, `썸네일 생성`, `저장`
- 썸네일 선택창: `다시 생성`, `선택`

### 보조 버튼 — 어두운 계열

- 메타데이터 편집창: `초기화`, `Try Parse On 141Jav`, `취소`
- 썸네일 선택창: `취소`

## 6. 창별 변경

### 메인 창

- 루트 `Window`에 기존 `CustomWindowStyle`을 적용한다.
- `CustomWindowStyle`의 타이틀바가 창의 `BorderBrush`를 배경으로 사용하므로 `MainWindow.BorderBrush`를 `#151828`로 지정한다.
- 기존 `Airi` 제목, 최소화·최대화·닫기 기능, 크기 조절과 최대화 동작은 유지한다.

### 메타데이터 편집창과 썸네일 선택창

- `SingleBorderWindow`를 `None`으로 바꿔 운영체제 타이틀바와 프레임을 제거한다.
- 기존 `CenterOwner`, `NoResize`, `ShowInTaskbar="False"`와 모달 소유 관계를 유지한다.
- 창 내부 제목, 취소 버튼과 기존 키보드 처리로 종료한다.
- 새 커스텀 타이틀바, 닫기 아이콘 또는 창 드래그 동작은 추가하지 않는다.

## 7. 변경 예상 위치

- `App.xaml`: 공통 작업 버튼 템플릿과 주요·보조 스타일
- `MainWindow.xaml`: 커스텀 타이틀바 적용 및 두 주요 작업 버튼 스타일 지정
- `Views/MetadataEditorWindow.xaml`: borderless 설정, 기존 로컬 버튼 스타일 제거, 역할별 공통 스타일 적용
- `Views/ThumbnailSelectionWindow.xaml`: borderless 설정 및 역할별 공통 스타일 적용
- `tests/Airi.Tests/`: 창 크롬과 버튼 역할 매핑을 검증하는 최소 WPF 테스트

코드 비하인드, ViewModel, 서비스와 도메인 모델은 변경하지 않는다.

## 8. 검증 전략

테스트를 먼저 추가해 현재 상태에서 다음 조건이 실패하는 것을 확인한 뒤 XAML을 수정한다.

- 메인 창이 `CustomWindowStyle`을 사용하고 `BorderBrush`가 `#151828`이다.
- 두 다이얼로그의 `WindowStyle`이 `None`이다.
- 대상 작업 버튼이 위 역할 분류에 맞는 공통 주요 또는 보조 스타일을 사용한다.
- 검색 지우기 버튼과 내부 컨트롤 버튼은 공통 작업 버튼 스타일의 대상이 아니다.

구현 후 다음 명령으로 전체 회귀와 빌드를 확인한다.

```powershell
dotnet test tests/Airi.Tests/Airi.Tests.csproj -c Debug
dotnet build Airi.sln -c Debug
graphify update .
```

## 9. 성공 기준

- 메인 창에 회색 네이티브 타이틀바 대신 앱 팔레트의 커스텀 타이틀바가 보인다.
- 두 다이얼로그에 운영체제 타이틀바와 프레임이 보이지 않는다.
- 대상 작업 버튼의 모서리, 테두리와 상태 표현이 일관된다.
- 주요 버튼은 파란색, 보조 버튼은 어두운 색으로 역할이 구분된다.
- 대상 밖의 버튼, 레이아웃과 기능 동작은 바뀌지 않는다.
- 신규 테스트, 기존 전체 테스트와 Debug 빌드가 통과한다.

## 10. 확정된 한계

- borderless 다이얼로그는 창 자체를 드래그할 수 없다. 두 창은 소유 창 중앙에 고정 크기로 열리고 기존 내부 동작으로 닫히므로 이번 범위에서는 드래그 기능을 추가하지 않는다.
- 메인 타이틀바는 프로젝트에 이미 포함된 `CustomWindowStyle`의 동작과 크기를 따른다. 별도 Windows 버전별 네이티브 크롬 조정은 추가하지 않는다.
- 미결정 항목: 없음
